1. INTRODUCTION                                              *tokencount-intro*

vim-tokencount displays a live, debounced token count for the current visual
selection and a one-shot count for any range or buffer. It pipes the text to
a long-lived Rust process running `tiktoken-rs` and reads the reply via an
asynchronous channel callback. Vim is never blocked: large selections,
whole-buffer counts, and very long files all return to the user immediately;
the count appears when the worker replies. A pure-vimscript approximation
is available for users who do not want to compile a binary.

==============================================================================
2. REQUIREMENTS                                       *tokencount-requirements*

  - Vim 9.1.1016 or newer (for `getregion()`, `str2blob()`, `noblock`)
  - +job and +timers
  - A Rust toolchain (`cargo`) at install time, OR `g:tokencount_fast = 1`

==============================================================================
3. STATUSLINE INTEGRATION                              *tokencount-statusline*

Embed the result of |TokenCountStatus()| anywhere in your 'statusline'. The
function returns an empty string outside visual mode, so wrap it in a `%(..%)`
group to suppress separators when nothing is selected: >

    set statusline+=\ %(\ %{TokenCountStatus()}%)

==============================================================================
4. CONFIGURATION                                            *tokencount-config*

                                                       *g:tokencount_executable*
Path to the compiled tokenizer binary. Defaults to the plugin's own
`target/release/tokencount`. Override to use a system-installed copy.

                                                      *g:tokencount_debounce_ms*
Milliseconds to wait after the last cursor move before sending a count
request. Default: 40.

                                                            *g:tokencount_label*
Prefix shown before the number. Default: `Tok:`. The count is approximate
for Claude models; see |tokencount-accuracy|.

                                                             *g:tokencount_fast*
If 1, skip the Rust binary entirely and report `chars / 3.5` as a fast
approximation. Default: 0.

==============================================================================
5. COMMANDS                                               *tokencount-commands*

                                                              *:TokenCount*
:[range]TokenCount      One-shot count of [range] (default: whole buffer).
                        Returns immediately and prints `Tok: ...` in the
                        message line. The real count is shown via |echom|
                        when the worker replies (also visible in
                        |:messages|). There is no size cap; whole-buffer
                        counts on very large files are supported.

                                                        *:TokenCountStatus*
:TokenCountStatus       Echo a health line: mode, binary path, job status.

==============================================================================
6. FUNCTIONS                                             *tokencount-functions*

                                                       *TokenCountStatus()*
TokenCountStatus()      Statusline-friendly string. Empty outside visual
                        mode, `Tok: N` inside. Updates asynchronously as
                        the selection changes; previous count remains
                        visible until the next reply lands.

==============================================================================
7. ACCURACY                                               *tokencount-accuracy*

The binary uses `tiktoken-rs` with OpenAI's `cl100k_base` tokenizer (the
GPT-4 / GPT-3.5-turbo tokenizer). It is not Claude's tokenizer. Anthropic
has never released an offline tokenizer, so the count is an approximation:

  - vs. GPT-4 / GPT-3.5:                exact
  - vs. Claude 3.x / 4.x (non-Opus 4.7): typically within 10-15%
  - vs. Claude Opus 4.7 (new tokenizer): can exceed 30% drift on code

Treat the count as an order-of-magnitude indicator, not a billing-grade
number.
